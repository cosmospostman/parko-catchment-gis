# Science Validation Tests

## Purpose

Science validation tests sit between unit tests and a research notebook. They
ask a specific question: **given real Sentinel-2 chip data at labelled presence
and absence points, do the signals described in OVERVIEW.md actually appear, and
do they discriminate?**

They are not software correctness tests. A failure does not mean the code is
broken — it means the signal is absent, too weak, or inconsistent with the
hypothesis. That outcome is scientifically meaningful and warrants investigation.

### The three test categories distinguished

| Category | Data | Assertion | Failure means |
|---|---|---|---|
| Unit | Synthetic | Logic is correct | Code bug |
| Integration | Synthetic / mocked | Pipeline stages compose correctly | Code bug |
| Science | Real Sentinel-2 chips | Expected signal is present and discriminative | Signal absent or analysis wrong |

## Location

```
tests/science/
  __init__.py
  conftest.py
  test_flowering_signal.py
  test_greenness_persistence.py
  test_peak_timing.py
  test_quality_weighting.py
```

Run separately from the main suite:

```bash
pytest tests/science/ -v
```

The main `pytest` run (`tests/unit/`, `tests/integration/`) never touches
`tests/science/` and always passes on a clean checkout.

## Data requirements

### Staged chip data

Science tests consume real chips fetched by `load-testdata`:

```
tests/fixtures/
  science_points.csv        # labelled points: point_id, lon, lat, label
  chips/                    # COG chips written by load-testdata
    {point_id}/
      {item_id}_{band}.tif
  .fixture_commit           # sentinel: HEAD commit when load-testdata ran
```

To stage:

```bash
python pipelines/train.py load-testdata
```

### Points CSV

`tests/fixtures/science_points.csv` is generated by `scripts/build_science_points.py`
from the pre-downloaded ALA gpkg. It is committed to the repo so `load-testdata`
can run without network access to ALA.

Current contents (generated 2026-04-04, seed=42):
- **40 presence points** — ALA-confirmed Parkinsonia, clustered near Longreach
  (145.04°E, 21.49°S), coordinate uncertainty ≤ 100m, ≥ 1 ALA neighbour within 50m
- **40 absence points** — regular 0.1° grid within S2 tile 55KCS
  (145.07–146.13°E, 21.79–20.79°S), ≥ 500m from any presence point.
  Constraining to one tile keeps all chips on the same STAC item set,
  making `load-testdata` fast.

Hard negatives (known Melaleuca/Acacia stands, `label=2`) are not yet included.
They can be added manually to the CSV; `load-testdata` will fetch chips for them
and science tests can filter by label.

## Skip guard

`tests/science/conftest.py` skips the entire module if chips are not staged:

```python
def pytest_configure(config):
    if not SENTINEL_FILE.exists():
        collect_ignore_glob.append("tests/science/*.py")
```

This means `pytest` without staged data completes cleanly — science tests are
simply not collected.

### Per-year discrimination

The three fixture years represent distinct rainfall regimes (2020 dry, 2021 typical, 2025 wet/recent).
`observations_by_year` enables per-cohort signal checks without pooling years together.

Typical usage in a test:

```python
def test_flowering_signal_by_year(observations_by_year, science_points):
    labels = science_points.set_index("point_id")["label"]
    for year, obs_by_point in observations_by_year.items():
        presence_peaks = [
            extract_waveform_features(obs, flowering_index, window=FLOWERING_WINDOW)["peak_value"]
            for pid, obs in obs_by_point.items()
            if labels[pid] == 1
            and extract_waveform_features(obs, flowering_index, window=FLOWERING_WINDOW)
        ]
        # assert or record per-year result into ReportCollector
```

Per-year results should be reported as separate sections in `report.md` so a weak signal in one year
is visible rather than averaged away. There is no pass/fail gate on individual years — the purpose
is to surface anomalies for investigation, not to assert expected outcomes we don't yet know.

Session-scoped so chip reads and extraction run once per `pytest tests/science/`
invocation regardless of how many test files import them.

## Diagnostic report

Each test run writes `tests/science/report.md` — a human-readable summary of
effect sizes, p-values, and per-signal verdicts. This is the primary output
when investigating signal quality or a failing hypothesis.

Format:

```
# Science Validation Report
Generated: 2026-04-03T15:30:00Z
Commit: abc1234
Points: 40 presence, 40 absence

## Signal 1: Flowering flush
Hypothesis: presence points have higher mean peak_value than absence (Aug–Oct)
  Presence mean:  0.43  (n=38)
  Absence mean:   0.21  (n=40)
  Mann-Whitney U: p=0.0003  ✓ PASS

## Signal 2: Dry-season greenness persistence
...
```

The report is written by a session-scoped autouse fixture in `conftest.py` that
collects results from all tests and writes at the end of the session. Individual
tests register their results into a shared `ReportCollector` object.

## Test specifications

### `test_flowering_signal.py`

**Hypothesis:** Presence points produce a higher quality-weighted flowering index
peak than absence points within the Aug–Oct window (DOY 213–304).

**Statistical test:** Mann-Whitney U (non-parametric; no normality assumption).

**Assertion:** `p < 0.05` AND median presence peak > median absence peak.

**Why Mann-Whitney:** Peak values are bounded [0, 1], likely skewed, small n.
A t-test would be inappropriate. Direction is separately asserted so a
significant result in the wrong direction also fails.

**What a failure means:** The flowering index is not detectable above noise at
this scale, or the Aug–Oct window is wrong for this location, or the ALA points
are contaminated (biocontrol defoliation, cleared stands).

---

### `test_greenness_persistence.py`

**Hypothesis:** Presence points have higher mean dry-season NDVI (July–September,
DOY 182–273) than absence points, and lower inter-annual variance.

Two sub-tests:
1. Mean dry-season NDVI: Mann-Whitney U, `p < 0.05`, presence > absence
2. Inter-annual NDVI variance: Mann-Whitney U, `p < 0.05`, presence variance < absence variance

**What a failure means:** The "stays green" signal is not distinguishable from
absence pixels at this location, or the absence points happen to include
phreatophytic vegetation (Melaleuca, riparian Eucalyptus).

---

### `test_peak_timing.py`

**Hypothesis:** Peak flowering DOY at presence points falls within the known
window for tropical QLD (Aug–Sep, DOY 213–273) and is more consistent
year-to-year than at absence points.

Two sub-tests:
1. Peak DOY distribution: chi-squared test of uniformity across months. Presence
   peaks should cluster in Aug–Sep (non-uniform), absence peaks should not
   show the same clustering.
2. Inter-annual SD of peak DOY: Mann-Whitney U, `p < 0.05`, presence SD < absence SD
   (calendar-locking signal from OVERVIEW.md).

**What a failure means:** The timing signal is absent or the inter-annual
consistency claim does not hold at this spatial cluster.

---

### `test_quality_weighting.py`

**Hypothesis:** Quality-weighted peak detection produces a lower false-peak rate
than raw maximum detection on the same observations.

**Method:** For each absence point, compare:
- Raw max of flowering index within window (unweighted)
- Quality-weighted peak (as per `extract_waveform_features`)

A "false peak" is defined as a peak value above `FLOWERING_THRESHOLD` that
occurs in a single low-quality observation (quality < 0.4). Count false peaks
under each method.

**Assertion:** Weighted method produces fewer false peaks than raw max
(one-sided binomial test or simple count comparison if n is small).

**What a failure means:** Quality weighting is not filtering the noise it is
designed to remove, or the threshold needs adjustment.

## What is NOT tested here

- **Code correctness** — that belongs in `tests/unit/`
- **Hard negatives** (Melaleuca, Acacia discrimination) — requires a species
  layer not yet assembled; add when `label=2` points are added to
  `science_points.csv`
- **Multi-year wet/dry robustness** — chips now cover 2020 (dry), 2021 (typical), 2025 (wet/recent); science tests can filter by year if needed
- **Absolute accuracy metrics** (AUC, F1) — these require a trained model and
  belong in the pipeline verify layer (`verify/`), not science tests

## Implementation order

1. `tests/science/conftest.py` — skip guard, shared fixtures, `ReportCollector`
2. `test_flowering_signal.py` — the primary discriminating signal; simplest to
   assert and most likely to show a clear effect
3. `test_greenness_persistence.py`
4. `test_peak_timing.py`
5. `test_quality_weighting.py`
6. Run `load-testdata`, execute suite, review `report.md`

## Relationship to `verify/`

`verify/` scripts run inline with `run.sh` after each pipeline step. They check
that a completed pipeline run produced plausible outputs for the current year's
data. Science tests run independently of the pipeline, against a small staged
fixture dataset, to validate the underlying assumptions before committing to a
full run.

```
science tests   →  validate assumptions   →  build confidence in approach
verify scripts  →  validate pipeline run  →  build confidence in outputs
```
